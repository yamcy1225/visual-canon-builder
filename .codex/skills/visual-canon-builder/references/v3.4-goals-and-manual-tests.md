# v3.4 Goals And Manual Tests

Use this reference when testing whether `$visual-canon-builder` preserves canon certainty, proportion safety, and `$imagegen` handoff discipline on realistic inputs.

## Registered Goals

1. Separate `observed` visual facts from `confirmed` canon.
2. Add an explicit `ready | provisional | blocked` handoff decision table.
3. Use per-landmark or per-envelope projection widths instead of one global character width.
4. Record yaw direction and subject-left/subject-right visibility for asymmetric details.
5. Replace generic `Constraints` with `Confirmed constraints`, `Provisional constraints`, and `Avoid`.
6. Use numeric tolerance only after calibrated reference-sheet measurements.
7. Give faction/world ontologies the same semantic mapping, provenance, and validation-shape structure as character ontologies.
8. Maintain manual golden tests with realistic image-derived cases.
9. Implement an interactive clarification loop with `question_queue`, user-answer provenance, and recomputed handoff state.
10. Avoid ping-pong questioning by producing provisional ontology and `$imagegen` handoff output immediately whenever unresolved questions block only `ready`, not `provisional`.
11. Add Evidence Interview RAG Mode so current-conversation evidence cards support every candidate assertion and batch approval locks final canon.

### Pass Criteria

- `Confirmed constraints` remains empty or minimal until linked assertions are explicitly approved with user_answers provenance.
- `Provisional constraints` contains observed but unapproved image-derived details.
- Every candidate assertion has at least one `evidence_refs` entry and `retrieval_scope: current_conversation_only`.
- Evidence cards include reviewable anchors such as `source_anchor`, `view_label`, and `bbox_or_region` or `source_text_span`.
- Before approval, every candidate assertion is `approval_status: pending_user_approval` or `keep_provisional`, not `approved`.
- `Approval Review Pack`, `Approval Payload`, and `Lock Summary` are present.
- `Handoff status` is `provisional`, not `ready`, while canon approval and permanent-prop status remain unresolved.
- `Handoff status` is not `blocked` when exactly one usable canon candidate exists and no identity-critical conflict exists.
- Ready-blocking questions appear in `question_queue` with stable IDs, affected fields, and separate provisional-blocking status.
- The skill does not stop for user input when unanswered questions block only a ready handoff; it proceeds with clearly labeled provisional output.
- User replies are represented as `user_answers` and linked back to questions/assertions before recomputation.
- Projection uses `derived_projected_widths` per landmark/envelope, not a single `derived_projected_width`.
- The skateboard action pose is marked scene-specific and low-confidence for measurement.
- Numeric tolerance is not applied before calibration: `calibration_status: uncalibrated` and `tolerance_applies: false`.
- Yaw direction and visible subject side are explicit in `view_spec`.
- `Text (verbatim)` is empty or explicitly provisional until shirt text is confirmed or requested as exact output text.
- The prompt pack uses `Confirmed constraints`, `Provisional constraints`, `Unresolved questions`, and `Avoid`; it does not use a generic `Constraints` field.
