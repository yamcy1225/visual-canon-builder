# v3.5 UX Manual Tests

Use these tests when checking that `$visual-canon-builder` remains ontology-safe while becoming easier for users to operate.

## Goals

1. Put `Guided Approval Interview`, `User Review`, `Quick Approval Table`, and `Next Reply Options` before technical ontology sections.
2. Hide hashes and long assertion IDs from the top-level user-facing prose.
3. Allow natural-language replies such as `추천대로 진행`, `전체 정본 승인`, `정체성과 의상은 승인, 소품은 임시`, and `수정: 긴 보드는 장면 소품`.
4. Preserve auditability by still emitting `Evidence Cards`, `Retrieval Trace`, `Approval Review Pack`, and `Approval Payload`.
5. Never promote unapproved candidate assertions into `Confirmed constraints`.

## Character Sheet First Pass

Input: one character sheet and a request to run Evidence Interview Mode.

Expected first-screen behavior:

```yaml
guided_approval_interview:
  status: awaiting_user_decision
  current_question_index: 1
  total_questions: 3
  current_question:
    label: "캐릭터 정체성"
    asks: "<identity approval question>"
    recommendation: "1"
    choices:
      - key: "1"
        label: "승인"
      - key: "2"
        label: "수정해서 승인"
      - key: "3"
        label: "임시 유지"
      - key: "4"
        label: "거절"

user_review:
  status: "초안 준비됨; 아직 정본 잠금 전"
  found:
    - "<character identity summary>"
    - "<outfit summary>"
    - "<prop or expression summary>"
  needs_decision:
    - label: "캐릭터 정체성"
      recommendation: approve
    - label: "기본 의상"
      recommendation: approve
    - label: "소품"
      recommendation: keep_provisional
  fast_reply:
    - "추천대로 진행"
    - "전체 정본 승인"
    - "정체성과 의상은 승인, 소품은 임시"
    - "수정: <항목>=<새 값>"
```

Expected technical behavior:

- The first visible approval section asks exactly one current question.
- Candidate assertions have `evidence_refs`.
- Initial candidate assertions are `pending_user_approval` or `keep_provisional`, not `approved`.
- `Confirmed constraints` is empty or contains only previously approved assertions.
- `$imagegen` handoff is `provisional` until approval arrives.

## One-By-One Approval

Input: user replies `1` to the first guided approval question.

Expected result:

- The current question is converted into a `user_answers` record.
- Affected assertions become `approval_status: approved` and `canon_status: confirmed`.
- The next response asks question 2/N instead of dumping only a full payload.
- Already answered questions appear under `answered`.

## Natural-Language Approval

Input: user replies `정체성과 의상은 승인, 소품은 임시`.

Expected result:

- Identity and outfit assertions get `user_answers` provenance and become `canon_status: confirmed`.
- Prop assertions remain `approval_status: keep_provisional`.
- `Confirmed constraints` includes only approved identity/outfit facts.
- `Provisional constraints` keeps the prop facts.
- `canon_lock_state` becomes `partially_locked` unless no pending canon-critical assertions remain.

## Revision Reply

Input: user replies `수정: 긴 보드는 장면 소품, 반바지 로고는 제거`.

Expected result:

- Original affected assertions become `approval_status: revised`.
- Replacement assertions are created with new IDs, versions, value hashes, and `user_answers` provenance.
- The prompt pack removes or provisionalizes the revised details according to the user answer.

## Audit Mode

Input: user asks for `strict mode`, `audit mode`, or `전체 YAML`.

Expected result:

- The output may show full `approval_payload` and all assertion IDs/hashes.
- The first section should still summarize the next required user action in plain language.
